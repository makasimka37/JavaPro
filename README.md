1. Загальний опис проєкту:
   
    Проєкт являє собою динамічний веб-додаток, створений для перегляду запланованих університетських заходів та реєстрації студентів на них. Головною особливістю системи є автоматичний контроль наявності вільних місць: додаток не дозволяє користувачам записуватися на події, де ліміт учасників уже вичерпано.

    Проєкт розроблено на базі архітектурного шаблону MVC (Model-View-Controller) із чітким розділенням коду на ізольовані шари, що полегшує його тестування та подальше розширення.


2. Технологічний стек:

    Мова програмування: Java 21 (використовуються сучасні класи для роботи з датою та часом).

    Веб-специфікації: Jakarta Servlet API 6.0 (обробка HTTP-запитів та управління життєвим циклом сервлетів).

    Технологія відображення: JSP (JavaServer Pages) — генерація динамічного HTML-контенту на стороні сервера.

    База даних: Реляційна СУБД PostgreSQL.

    Доступ до даних: Чистий JDBC (Java Database Connectivity) без використання важких ORM-фреймворків.

    Збірка проєкту: Apache Maven (керування залежностями та пакування проєкту у веб-архів WAR).

    Тестування: JUnit Jupiter 5.10.2 (написання ізольованих модульних тестів).

    Сервер застосунків: Apache Tomcat 10.1 (забезпечує підтримку специфікацій Jakarta).


3. Архітектура та структура проєкту:
   
    Проєкт розділено на шість основних логічних пакетів, кожен з яких відповідає за свою унікальну задачу:

    Пакет model (Шар моделей даних): Містить POJO-класи (сутності Event та Participant), які відображають структуру таблиць бази даних у Java-об'єкти. Поля класів інкапсульовані та доступні лише через методи геттери та сеттери.

    Пакет database (Шар конфігурації БД): Містить єдину точку підключення до СУБД — клас DatabaseContext. Він ініціалізує JDBC-драйвер PostgreSQL у статичному блоці та повертає активні об'єкти з'єднання Connection.

    Пакет repository (Шар доступу до даних / DAO): Класи EventRepository та ParticipantRepository. Вони містять чистий SQL-код і відповідають за виконання запитів до бази даних (вибірка майбутніх подій, пошук конкретного заходу за його ідентифікатором та збереження нових учасників).

    Пакет service (Шар бізнес-логіки): Клас EventService. Це "мозок" програми, де описано головне бізнес-правило: перевірка наявності вільних місць перед остаточним внесенням студента до списку учасників. Шар повністю ізольований від SQL та HTML коду.

    Пакет servlet (Шар контролерів / Веб-рівень): Класи EventListServlet (обробляє GET-запити для виведення головної сторінки) та RegisterServlet (обробляє POST-запити при надсиланні форми реєстрації).

    Папка WEB-INF/jsp (Шар відображення / View): Файл list.jsp. Він містить HTML-розметку із вбудованими Java-скриптлетами для динамічного відображення таблиці заходів та форми запису. Папка схована всередині структури WEB-INF, що унеможливлює прямий доступ користувачів до сторінки в обхід сервлетів.


4. Опис структури Бази Даних:
   
    Система використовує дві реляційні таблиці, пов'язані між собою зв'язком «один-до-багатьох» (один захід може мати багато зареєстрованих учасників):

    
Таблиця events (Заходи):

    id — первинний ключ, ціле число з автоматичним збільшенням (SERIAL).

    title — текстова назва заходу.

    event_date — дата та час проведення (TIMESTAMP).

    max_seats — максимальний ліміт місць у залі.
Таблиця participants (Учасники):

    id — первинний ключ.

    event_id — зовнішній ключ (FOREIGN KEY), який посилається на ідентифікатор заходу з таблиці events. Для забезпечення цілісності даних налаштовано правило каскадного видалення (ON DELETE CASCADE): якщо захід видаляється, всі його учасники видаляються автоматично.

    student_name — повне ім'я студента.

    student_email — контактна електронна адреса.

5. Ключові технічні рішення та патерни:

Безпека та захист від SQL-ін'єкцій:

Для виконання всіх запитів до бази даних у репозиторіях використовуються виключно параметризовані об'єкти PreparedStatement замість звичайного Statement. Дані від користувача підставляються замість знаків питання як текстові параметри, що унеможливлює впровадження стороннього шкідливого SQL-коду.


Автоматичне керування ресурсами:

Усі операції з базою даних обгорнуті в конструкцію try-with-resources. Об'єкти підключення Connection та команди PreparedStatement автоматично закриваються середовищем Java після завершення роботи методу, що повністю виключає ризик витоку пам'яті та переповнення пулу з'єднань СУБД.

Патерн PRG (Post/Redirect/Get):

Для обробки форми реєстрації в RegisterServlet застосовано патерн PRG. Після успішного опрацювання POST-запиту від форми, контролер не перенаправляє користувача напряму через forward, а робить повноцінний редирект (sendRedirect) на головну сторінку. Це захищає систему від випадкового дублювання даних у базі, якщо користувач вирішить оновити сторінку натисканням клавіші F5 у браузері.

Динамічний інтерфейс користувача:

На сторінці list.jsp реалізовано логіку перевірки залишку місць за допомогою тернарних операторів Java. Якщо кількість вільних місць на конкретний захід дорівнює нулю, система автоматично додає атрибут disabled до відповідного пункту у випадаючому списку форми. Це блокує можливість обрання заповненого заходу на рівні інтерфейсу. Додатково в URL-адресу передаються маркери успіху або помилки, на основі яких на екран виводяться інформаційні блоки для користувача.

Модульне JUnit тестування:

Клас бізнес-логіки EventService покритий ізольованим модульним тестом EventServiceTest. Для того, щоб тест перевіряв виключно алгоритми Java і не залежав від реальної бази даних чи наявності підключення до неї, було використано патерн проектування Dependency Injection (Впровадження залежностей) через конструктор. У тесті створюються анонімні класи-заглушки (Stubs/Mocks) для репозиторій-шарів, які імітують поведінку бази даних та примусово повертають тестові об'єкти подій із нульовою кількістю вільних місць. Тест перевіряє, що в такому випадку сервіс коректно відхиляє реєстрацію та повертає значення false.
